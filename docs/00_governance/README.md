# Governance

This folder defines the **non‑negotiable governance rules** for the
Kant‑Mak‑Um Project Overview repository and its surrounding GitHub ecosystem.

Governance documents change **rarely**.  
They exist to establish clarity, boundaries, and authority—not to track daily work.

---

## Purpose of Governance

Governance answers four questions definitively:

1. **What belongs in this repository**
2. **What explicitly does not belong in this repository**
3. **How repositories relate to each other**
4. **Where authority lives when boundaries are unclear**

If there is ever uncertainty, governance documents take precedence.

---

## Scope of This Repository

The `kant-mak-um-project-overview` repository is the **coordination core** of the
Kant‑Mak‑Um ecosystem.

It exists to define:
- Repository roles and boundaries
- Open vs closed IP separation
- Naming and structural conventions
- Academic integration (MS‑ECE, ME‑EM)
- Operating rhythm and decision discipline

This repository **does not execute projects**.

---

## Explicit Exclusions

This repository does **not** contain:

- Product CAD or configurations
- Manufacturing processes or tooling
- ORCAS motor internals
- Coconino arm internals
- AFP process IP
- Brand‑specific execution artifacts
- Website content or deployment code

Those belong in their respective repositories, under their own licenses
and access controls.

---

## Authority Model

The Kant‑Mak‑Um GitHub ecosystem is **tiered**, not flat.

Authority is distributed as follows:

- **Coordination Core (this repo)**  
  Structural rules, interfaces, and system‑level clarity

- **Open Design Commons**  
  Public CAD and configuration IP

- **Shared Components**  
  Reusable open components across brands

- **Manufacturing Core (Cellular Composites)**  
  Proprietary processes, tooling, and manufacturing IP

- **Product Company Repos**  
  Brand‑specific products, configurations, and commercial execution

No repository may assume authority outside its tier.

---

## Open vs Closed IP Boundary

This ecosystem is intentionally hybrid:

- **Design intent is open**
- **Manufacturing execution is closed**

This boundary is foundational and must not be blurred.

If a question arises about whether something is:
- *design* → it is likely open  
- *process* → it is likely closed  

Governance documents define and enforce this separation.

---

## Change Discipline

Governance documents:
- Change infrequently
- Are revised deliberately
- Are updated only during review cycles

Ad‑hoc edits are discouraged.

When governance changes, the reason for change should be recorded.

## Guiding Principle

> **Clarity beats cleverness.  
> Boundaries beat convenience.  
> Structure enables freedom.**

This governance exists so the ecosystem can grow without collapsing
under its own complexity.
